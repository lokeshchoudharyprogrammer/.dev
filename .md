## 1. Vertical Scaling

### What it is
Increasing the capacity of a **single machine** — more CPU cores, more RAM, faster storage (SSD → NVMe), or a higher-bandwidth network interface. The code and architecture stay the same; you just run it on a bigger box.

### Why you'd use it
- No code changes required — fastest path to more capacity
- No distributed state management
- Great for stateful components that are hard to shard (e.g., primary databases)
- Predictable performance — no network hops between nodes

### How it works
You request a larger instance from your cloud provider (e.g., upgrading an AWS RDS `db.t3.medium` → `db.r6g.4xlarge`). The OS and application see more resources and automatically use them (more parallel threads, larger memory heap, bigger buffer pools).

### Trade-offs

| Pros | Cons |
|------|------|
| Simple — no architectural changes | Hard ceiling (biggest instance available) |
| No distributed system complexity | Single point of failure remains |
| Useful for databases, caches, monoliths | Expensive at high tiers |
| Immediate performance gain | Requires downtime during upgrade (often) |

### When to stop vertical scaling
When the cost per unit of performance starts rising steeply, when you hit the largest available instance, or when a single machine still cannot handle peak load — that's the signal to move to horizontal scaling.

### Real-world signals
- AWS EC2 instance family upgrades (t3 → m6i → r6i → x2idn)
- PostgreSQL primary DB RAM upgrade to increase shared_buffers
- Redis in-memory store scaled up to fit the full working set

### Key talking point
> "Vertical scaling buys time cheaply, but it doesn't solve fault tolerance. A bigger single machine is still a single point of failure."

## 2. Preprocessing

### What it is
Performing **expensive or time-consuming tasks ahead of time** — during low-traffic windows — so that the work is already done when a user request arrives. Instead of computing something at request time, you pre-compute it and serve the cached result.

### Why you'd use it
- Dramatically reduces response latency for expensive operations
- Smooths load on the system — spreads CPU/IO over a longer time window
- Isolates heavy batch work from user-facing traffic
- Enables serving pre-built results instantly (reports, thumbnails, recommendations)

### How it works
A scheduler (cron job, Airflow DAG, AWS Lambda on a schedule) triggers a job during off-peak hours (e.g., 2–5 AM). The job runs its computation and stores results in a fast-access store (cache, database, CDN). When users request the data, the result is already there.

### Common patterns

**Cron jobs** — time-triggered shell/Python scripts for simple recurring tasks.

**Batch pipelines** — frameworks like Apache Spark, Flink, or AWS Glue for large-scale data transformation.

**Pre-warming caches** — populate Redis or Memcached before peak traffic hits.

**Pre-generating static assets** — build HTML pages, PDFs, or images before they're requested (Jamstack / static site generation).

**Recommendation pre-computation** — run ML inference nightly and store top-N recommendations per user in a database.

### Trade-offs

| Pros | Cons |
|------|------|
| Fast reads — result already exists | Data staleness between runs |
| Reduces peak-time load | Storage cost for pre-computed results |
| Predictable job scheduling | Job failures can leave stale or missing data |
| Decouples heavy computation from user requests | Not suitable for real-time or user-specific dynamic data |

### Staleness tolerance question
Always ask: **"How fresh does this data need to be?"**
- Analytics dashboards — hourly or nightly is fine
- Product inventory — needs near real-time
- Live prices / sports scores — must be real-time (preprocessing not appropriate)

### Key talking point
> "Preprocessing trades storage space and data freshness for speed. It's one of the simplest and most effective performance optimisations when read patterns are predictable."

## 3. Backup Servers

### What it is
Running **redundant copies of a resource** so that if one fails, another can take over — eliminating single points of failure (SPOF).

### Why you'd use it
Any component that exists as only one instance is a SPOF. If it crashes, your entire system goes down. Backup servers ensure continuity.

### The two main models

#### Active-Active
Both servers handle live traffic simultaneously. If one fails, the other absorbs all traffic with no interruption.
- Zero downtime on failure
- Full capacity during failure (since both were already serving)
- Requires state synchronisation between nodes

#### Active-Passive (Primary-Standby)
One server handles all traffic (primary); the other stays idle and ready (standby/replica). On primary failure, the standby is promoted.
- Brief failover time (detection + promotion, usually seconds to minutes)
- Simpler — no need to sync live writes across two active nodes
- Lower cost (standby can be a smaller instance)

### Where backup servers apply
- **Web/app servers** — multiple instances behind a load balancer (effectively active-active)
- **Databases** — primary + read replicas; automated failover (AWS RDS Multi-AZ, PostgreSQL streaming replication)
- **Load balancers themselves** — run a secondary LB for LB-level redundancy
- **DNS** — multiple nameservers
- **Message brokers** — Kafka replicas across brokers
- **Power supplies** — dual PSUs in physical servers

### Key metrics affected
- **RTO (Recovery Time Objective)** — how long can the system be down? Active-active = near zero. Active-passive = minutes.
- **RPO (Recovery Point Objective)** — how much data can be lost? Synchronous replication = zero. Asynchronous = seconds of lag.
- **MTTR (Mean Time To Recovery)** — backup servers directly reduce this

### Trade-offs

| Pros | Cons |
|------|------|
| Eliminates SPOF | Higher infrastructure cost |
| Reduces MTTR dramatically | Consistency challenges (active-active) |
| Enables rolling updates with zero downtime | More complex monitoring and failover logic |

### Key talking point
> "In any system design interview, you should identify every SPOF and propose a redundancy strategy. There is no excuse for a single database with no replica in a production system."

## 4. Horizontal Scaling

### What it is
Adding **more machines of the same type** to handle increased load, instead of making one machine bigger. Traffic is split across all instances.

### Why you'd use it
- Unlimited scale ceiling — just keep adding nodes
- Cost-effective — many commodity machines vs one expensive server
- Inherent redundancy — losing one node doesn't take down the system
- Independent scaling of each service tier

### How it works
Multiple identical copies of your service run in parallel. A load balancer distributes incoming requests across them. Adding capacity means deploying a new instance (manually or automatically via auto-scaling).

### The critical prerequisite: Stateless services
For horizontal scaling to work, **each instance must be able to handle any request** — it cannot hold state in local memory that other instances don't have.

**State must move to shared external stores:**
- **Sessions** → Redis or a database (not in-process memory)
- **File uploads** → S3 or a shared filesystem (not local disk)
- **Cache** → Redis or Memcached (not a local LRU cache)

### Auto-scaling
Cloud auto-scaling groups monitor a metric (CPU %, request queue depth, custom metrics) and automatically add/remove instances:
- Scale out when CPU > 70% for 3 minutes
- Scale in when CPU < 30% for 10 minutes
- Prevent thundering-herd: always keep a minimum instance count

### Trade-offs

| Pros | Cons |
|------|------|
| Virtually unlimited scale | Requires stateless architecture |
| Cost-efficient at scale | Adds deployment/orchestration complexity |
| Fault-tolerant by design | Distributed state consistency challenges |
| Supports zero-downtime deploys (rolling updates) | Network latency between nodes |

### Tools
- Kubernetes (HPA — Horizontal Pod Autoscaler)
- AWS Auto Scaling Groups
- Google Cloud Managed Instance Groups

### Key talking point
> "Horizontal scaling is the right long-term answer for web-tier and application-tier services, but it requires you to externalise all state first. If you haven't done that, horizontal scaling just creates worse bugs."

## 5. Microservices

### What it is
An architectural style where a system is decomposed into **small, independently deployable services**, each responsible for a specific business capability and owning its own data.

### Why you'd use it
- Teams can deploy independently — no coordinated release trains
- Each service can be scaled independently based on its own load
- Technology heterogeneity — each service can use the best language/framework for its job
- Isolated failures — a crash in the payments service doesn't crash the catalogue service

### Contrast with a monolith
In a monolith, all functionality lives in one deployable unit. Advantages: simpler local dev, easier transactions, no network overhead between components. Disadvantages: slow build/deploy cycles, scaling is all-or-nothing, a bug in one module can crash everything.

### Decomposition strategies

**By business capability** (preferred) — one service per domain: `UserService`, `OrderService`, `PaymentService`, `InventoryService`. Aligns with Conway's Law — team structure mirrors the service structure.

**By bounded context (DDD)** — Domain-Driven Design identifies the natural seams in a business domain. Each bounded context becomes a service.

**Strangler Fig pattern** — incrementally extract microservices from a monolith by routing specific endpoints to new services while the monolith handles the rest.

### Inter-service communication

**Synchronous (REST / gRPC)** — Service A calls Service B and waits. Simple, but creates temporal coupling: if B is slow or down, A is affected.

**Asynchronous (events/messages)** — Service A publishes an event to a queue; Service B consumes it independently. More resilient, but harder to trace and reason about.

### Data isolation
Each microservice owns its own database — no shared tables. This enforces independence but means cross-service queries require API calls or event-driven data replication, not SQL JOINs.

### Trade-offs

| Pros | Cons |
|------|------|
| Independent deployments | Network latency between services |
| Independent scaling per service | Distributed tracing complexity |
| Polyglot — best tool for each job | Data consistency across service boundaries |
| Isolated failures | Operational overhead (N services to monitor/deploy) |
| Small, focused codebases | Testing distributed interactions is hard |

### Anti-patterns to avoid
- **Nano-services** — splitting too finely creates more overhead than value. A service that handles only one DB table is almost certainly too small.
- **Distributed monolith** — services that must always deploy together or that share a database. You get the complexity of microservices with none of the benefits.
- **Chatty services** — a user-facing API that fans out to 15 synchronous service calls to render a single page. Use an API gateway with aggregation or a BFF (Backend for Frontend) pattern.

### Key talking point
> "Microservices are an organisational scaling pattern as much as a technical one. If a single team owns all the services, you probably don't need them yet. Start with a well-structured monolith and extract services as team size and deployment frequency demand it."

## 6. Distributed Systems

### What it is
A collection of **independent computing nodes** that communicate over a network and coordinate to appear as a single coherent system to users.

### Why it exists
No single machine is fast enough, reliable enough, or geographically close enough to all users. Distributing across many nodes provides:
- Higher throughput (parallelism)
- Fault tolerance (redundancy)
- Lower latency (geographic distribution)
- Scale beyond any single machine's limits

### The CAP Theorem
In the event of a **network partition** (nodes cannot communicate), you must choose:

- **Consistency (C)** — every read returns the most recent write. All nodes agree on the same data at all times.
- **Availability (A)** — every request receives a response (not an error), even if it might be stale.
- **Partition Tolerance (P)** — the system continues operating despite network failures between nodes.

**P is not optional** — networks fail. So the real choice is: **CP or AP**?

| System | Choice | Reasoning |
|--------|--------|-----------|
| Banking / payments | CP | Stale balance is catastrophic |
| DNS | AP | Stale records are tolerable; always responding is critical |
| Cassandra | AP (tunable) | Prioritises availability; eventual consistency |
| HBase / Zookeeper | CP | Consistency over availability |

### PACELC (extends CAP)
Even when there is **no partition**, you must trade off **latency vs consistency**:
- Lower latency → accept some inconsistency (async replication)
- Perfect consistency → accept higher latency (synchronous replication, consensus rounds)

### Consensus algorithms
When distributed nodes must agree on a value (which node is leader, what the next log entry is):
- **Raft** — leader election + log replication. Easier to understand than Paxos. Used in etcd, CockroachDB, TiKV.
- **Paxos** — the original consensus algorithm. Proven correct, but notoriously difficult to implement correctly.

### Failure modes in distributed systems

| Failure | Description |
|---------|-------------|
| Node crash | A machine stops responding |
| Network partition | Two groups of nodes cannot reach each other |
| Clock skew | Different clocks on different nodes drift apart |
| Byzantine fault | A node sends incorrect or malicious data |
| Partial failure | Some operations succeed, others fail mid-transaction |

### Key distributed systems concepts

**Eventual consistency** — given no new writes, all nodes will eventually converge to the same value. Used in DNS, Cassandra, DynamoDB (default).

**Strong consistency** — a read always reflects the latest write. Requires coordination overhead (locks, consensus). Used in Spanner, Zookeeper.

**Replication** — copies of data on multiple nodes for fault tolerance and read scaling. Can be synchronous (higher consistency, higher latency) or asynchronous (lower latency, risk of data loss on failure).

**Sharding** — splitting data horizontally across nodes by a shard key (e.g., user_id % N). Each node owns a subset of the data. Increases write throughput but complicates cross-shard queries.

**Leader election** — one node is designated to coordinate writes. Common pattern: Raft leader, Redis Sentinel, Kafka controller.

### Key talking point
> "Distributed systems trade simplicity for scale and resilience. Before distributing anything, ask whether the problem actually requires it — a single well-tuned PostgreSQL instance handles millions of requests per day for many production workloads."

## 7. Load Balancing

### What it is
A **load balancer** sits in front of a pool of servers and routes each incoming request to the most appropriate backend, distributing work evenly and removing unhealthy nodes from rotation.

### Why you need it
Without a load balancer, clients would need to know the address of every server — and one overloaded server cannot share its burden with a neighbour. The load balancer is the single entry point that provides:
- Even traffic distribution
- Automatic failover when a server goes down
- TLS termination (decrypts HTTPS once, forwards plaintext internally)
- A stable endpoint for clients despite backend changes

### Layer 4 vs Layer 7

**Layer 4 (Transport layer)** — routes based on IP address and TCP/UDP port. Very fast (minimal inspection), but cannot make routing decisions based on request content.
- Use when: pure TCP throughput, non-HTTP protocols (database connections, game servers)

**Layer 7 (Application layer)** — inspects the full HTTP request (URL path, headers, cookies, body). Can make intelligent routing decisions:
- Route `/api/*` to API servers, `/static/*` to CDN
- Sticky sessions via cookie inspection
- A/B testing by routing % of traffic to a new version
- Use when: HTTP/HTTPS traffic, content-based routing, WebSockets

### Load balancing algorithms

| Algorithm | How it works | Best for |
|-----------|-------------|----------|
| Round-robin | Each server in turn | Uniform request cost |
| Weighted round-robin | More requests to faster/bigger servers | Heterogeneous server pool |
| Least connections | Route to server with fewest active connections | Long-lived connections (WebSockets, streaming) |
| IP hash | Hash client IP → consistent server | Stateful sessions without shared session store |
| Random | Pick a random server | Simple, surprisingly effective at scale |
| Least response time | Route to fastest-responding server | Latency-sensitive applications |

### Health checks
The load balancer periodically sends a probe (HTTP GET `/health`, TCP connect, or ICMP ping) to each backend. If a backend fails N consecutive checks, it is removed from rotation. When it recovers, it is re-added. This is **automatic failure detection without human intervention**.

### Session persistence (sticky sessions)
Some applications require all requests from the same user to go to the same server (e.g., in-memory session state). The LB uses a cookie or IP hash to "stick" a client to one backend. Note: sticky sessions undermine even distribution and create a SPOF for that user if the server crashes — better to externalise session state and avoid stickiness.

### Where load balancers live
- **DNS-level** — multiple A records, clients round-robin (coarse, slow to update)
- **Hardware** (F5, Citrix ADC) — high throughput, expensive
- **Software** (Nginx, HAProxy) — flexible, free, widely used
- **Cloud-managed** (AWS ALB/NLB, GCP Load Balancer, Cloudflare) — zero-ops, auto-scales

### Key talking point
> "A load balancer also needs to be redundant — an LB with no backup is itself a SPOF. Production setups run active-passive LB pairs or use managed cloud LBs that are redundant by default."

## 8. Decoupling

### What it is
Designing components so they are **independent** — they do not call each other directly, do not share state, and do not need to know about each other's internals. Changes in one component do not force changes in others.

### Why it matters
Tightly coupled systems become fragile: a slow dependency makes your service slow; a crashed dependency crashes your service; deploying one component requires coordinating with others. Decoupling breaks these chains.

### Tight coupling vs loose coupling

**Tight (synchronous REST call):**
Service A calls Service B's HTTP endpoint and blocks waiting for a response. If B is slow, A is slow. If B crashes, A crashes. A must know B's address and API contract.

**Loose (async message queue):**
Service A publishes an event to a queue. Service B consumes from the queue at its own pace. A doesn't know B exists. B can be restarted, upgraded, or replaced without A caring.

### Tools for decoupling

**Message queues** — point-to-point. One producer, one consumer. Example: SQS, RabbitMQ. Use for work queues (process one job exactly once).

**Publish-Subscribe (Pub/Sub)** — one producer, many consumers. Each consumer gets a copy of every message. Example: Kafka, SNS, Google Pub/Sub. Use for events that multiple systems care about (order placed → notify inventory, email service, analytics, fraud detection).

**Event-driven architecture** — services communicate only via events. No service ever calls another directly. Maximally decoupled, but harder to trace the flow of a request.

**API Gateway** — provides a stable contract to clients even as the backend services evolve. Clients decouple from the physical location and versioning of services.

### Decoupling dimensions

| Dimension | What it means |
|-----------|---------------|
| Temporal | Producer and consumer don't need to be running at the same time |
| Spatial | Producer doesn't need to know consumer's address |
| Synchronisation | Producer doesn't wait for consumer to finish |

### Trade-offs

| Pros | Cons |
|------|------|
| Services can scale independently | Harder to trace request flow (need distributed tracing) |
| Failure isolation — B going down doesn't crash A | Eventual consistency — B processes event later |
| Independent deployment | More infrastructure (queues, brokers) |
| Easier to add new consumers without changing producer | Debugging distributed async flows is complex |

### Key talking point
> "Every time you introduce a direct synchronous call between services, you create a dependency chain. A message queue between them is almost always worth the tradeoff for non-trivial systems."

## 9. Logging and Metrics

### What it is
**Observability** — the ability to understand what your system is doing from the outside. It has three pillars: **logs** (events), **metrics** (numbers over time), and **traces** (request journeys).

Without observability, you are operating blind: you cannot diagnose incidents, prove SLAs, or detect degradation before it becomes an outage.

---

### Pillar 1: Logs

**What they are:** Timestamped, human-readable records of discrete events.

**Best practices:**
- Use **structured logging** (JSON) so logs are machine-parseable and searchable
- Include: timestamp, log level (DEBUG/INFO/WARN/ERROR), service name, trace ID, user ID (if applicable), message, and context fields
- **Always include a trace/correlation ID** so you can follow one request across 10 microservices
- Set appropriate log levels — don't log DEBUG in production at full volume (cost and noise), but ensure WARN and ERROR are always captured
- Logs should be **immutable and append-only** — never modify a log after the fact

**Example structured log entry:**
```json
{
  "timestamp": "2025-04-06T14:32:01Z",
  "level": "ERROR",
  "service": "payment-service",
  "trace_id": "abc-123-xyz",
  "user_id": "u_9872",
  "message": "Payment gateway timeout",
  "duration_ms": 5003,
  "gateway": "stripe"
}
```

**Tools:** ELK Stack (Elasticsearch + Logstash + Kibana), Grafana Loki, AWS CloudWatch Logs, Datadog Logs, Splunk.

---

### Pillar 2: Metrics

**What they are:** Numeric time-series data sampled at intervals. Aggregated and stored efficiently for graphing and alerting.

**Key monitoring methodologies:**

**USE Method** (for infrastructure / resources):
- **U**tilisation — what % of capacity is being used?
- **S**aturation — is work queuing up because the resource is full?
- **E**rrors — are there hardware or OS-level errors?

**RED Method** (for services / request-based systems):
- **R**ate — how many requests/second is the service handling?
- **E**rrors — what % of requests are failing?
- **D**uration — how long are requests taking (p50, p95, p99)?

**The Four Golden Signals** (Google SRE):
1. Latency (time to serve a request)
2. Traffic (demand on the system)
3. Errors (rate of failed requests)
4. Saturation (how "full" the service is)

**Key metrics to track:**
- HTTP request rate, error rate, and latency (p50/p95/p99)
- CPU utilisation, memory usage, disk I/O, network I/O
- Queue depth (growing queue = consumer can't keep up)
- Database connection pool usage, query latency
- Cache hit rate

**Tools:** Prometheus + Grafana, Datadog, New Relic, AWS CloudWatch, InfluxDB.

---

### Pillar 3: Distributed Tracing

**What it is:** A trace records the full journey of a single request across multiple services, showing exactly where time was spent.

**Why it matters:** In a microservices architecture, a single user request might touch 10 services. Logs and metrics tell you something is wrong; traces tell you *exactly which service and which database call* is causing the latency.

**How it works:** A unique trace ID is generated at the entry point and propagated in request headers (e.g., `X-Trace-Id`). Each service creates a **span** (start time, end time, service name, operation). Spans are reported to a central backend that assembles them into a trace tree.

**Tools:** Jaeger, Zipkin, AWS X-Ray, Honeycomb, OpenTelemetry (vendor-neutral instrumentation standard).

---

### Alerting best practices
- Alert on **symptoms** (user-facing latency high, error rate high), not causes (CPU high)
- Use **SLOs (Service Level Objectives)** and **error budgets** to decide when to alert
- Avoid alert fatigue — every alert must be actionable
- Runbooks: every alert should link to a document explaining what to do

### Key talking point
> "Logs answer 'what happened', metrics answer 'how is the system performing right now', and traces answer 'why is this specific request slow'. You need all three. Without them, you're debugging in production by guessing."

## 10. Extensibility

### What it is
Building systems that are **easy to modify and extend** without requiring large rewrites. New features are added by writing new code, not by changing existing code.

### Why it matters
Requirements always change. A system built without extensibility in mind becomes a liability: every new feature requires risky changes to existing, tested code. Extensible systems absorb change cheaply.

### The Open/Closed Principle (from SOLID)
> "Software entities should be **open for extension** but **closed for modification**."

You add new behavior by writing new code (a new class, a new plugin, a new handler), not by editing existing code. This preserves the correctness of existing code and reduces regression risk.

**Violation example (NOT extensible):**
```python
def calculate_discount(user_type):
    if user_type == "student":
        return 0.10
    elif user_type == "senior":
        return 0.15
    elif user_type == "employee":  # adding this requires editing the function
        return 0.20
```

**Extensible version (Strategy pattern):**
```python
class DiscountStrategy:
    def calculate(self, price): ...

class StudentDiscount(DiscountStrategy):
    def calculate(self, price): return price * 0.10

class SeniorDiscount(DiscountStrategy):
    def calculate(self, price): return price * 0.15

# Adding EmployeeDiscount requires NO changes to existing code
```

### Patterns that enable extensibility

**Strategy pattern** — inject different algorithms at runtime. Open/Closed by design.

**Plugin / hook architecture** — expose lifecycle hooks that plugins can register against. The core system never changes; new behavior is added via plugins.

**Event-driven hooks** — emit events at key points; listeners can be added without modifying the emitter.

**Feature flags** — enable/disable features at runtime without deployment. Roll out to 1% of users, test, expand. No code changes.

**Dependency injection** — pass dependencies in rather than hardcoding them. Makes it trivial to swap implementations (e.g., swap a real payment gateway for a test double).

**Interface-based design** — depend on abstractions (interfaces), not concrete implementations. New implementations can be plugged in without changing callers.

### Signals of poor extensibility
- Long `if/elif/switch` chains that must be edited for every new case
- Hardcoded class names, URLs, or configuration values in business logic
- No interfaces — everything depends on concrete types
- A "change one thing, test everything" deployment cycle

### When NOT to over-extend
> "Premature abstraction is the root of much software evil."

Don't add extension points for requirements that don't exist yet. YAGNI (You Aren't Gonna Need It). Add extensibility at the seams you **know** will change — not everywhere speculatively.

### Key talking point
> "Extensibility is about predicting the axes of change in your system and designing them to be cheap to vary. Every plugin, strategy, and feature flag is a bet that you've identified the right axis of change."

## 11. Low-Level System Design

### What it is
The practice of writing **efficient, clean, maintainable code** at the class, object, and function level. High-level architecture is irrelevant if the code it runs is poorly structured, inefficient, or unmaintainable.

---

### SOLID Principles

| Principle | Full Name | Core idea |
|-----------|-----------|-----------|
| **S** | Single Responsibility | A class/function should have one reason to change |
| **O** | Open/Closed | Open for extension, closed for modification |
| **L** | Liskov Substitution | Subclasses must be usable wherever the base class is used |
| **I** | Interface Segregation | Clients should not depend on methods they don't use |
| **D** | Dependency Inversion | Depend on abstractions, not concrete implementations |

---

### Design Patterns

**Creational** — how objects are created:
- **Singleton** — exactly one instance of a class. Use for configuration managers, connection pools, loggers. Risk: global state, hard to test.
- **Factory** — a method or class that creates objects without specifying the exact class. Use when creation logic is complex or the concrete type varies.
- **Builder** — construct complex objects step by step. Use for objects with many optional fields (avoids telescoping constructors).

**Structural** — how classes/objects are composed:
- **Adapter** — converts one interface to another. Use when integrating third-party libraries or legacy code.
- **Decorator** — add behaviour to an object dynamically without subclassing. Use for logging, authentication, caching wrappers.
- **Proxy** — a surrogate object that controls access to another object. Use for lazy loading, access control, remote service calls.

**Behavioural** — how objects communicate:
- **Observer** — objects subscribe to events from a subject. Use for event-driven UIs, notifications, pub/sub.
- **Strategy** — inject interchangeable algorithms. Use for sorting, pricing, authentication.
- **Command** — encapsulate a request as an object. Use for undo/redo, queued operations, audit logs.

---

### Data Structures and Algorithms — choosing right

| Need | Use | Why |
|------|-----|-----|
| O(1) lookup by key | HashMap / HashSet | Direct addressing |
| Ordered iteration + O(log n) ops | TreeMap / SortedSet | Binary search tree |
| Queue/deque operations | LinkedList / ArrayDeque | O(1) head/tail ops |
| Priority-based processing | Heap (PriorityQueue) | O(log n) insert/remove-min |
| Frequent range queries | Sorted array + binary search | Cache-friendly |
| Graph shortest path | Dijkstra (weighted) / BFS (unweighted) | —  |

**The most common mistake:** using a `List` for membership checks (`if x in list`) — O(n) per check — when a `Set` would be O(1).

---

### Code-level performance

**Avoid N+1 queries** — the most common database performance killer. Fetching a list of 100 users then querying the DB once per user = 101 queries. Use JOINs or batch queries instead.

**Index your database columns** — a full table scan on a large table costs O(n); an index lookup costs O(log n). Add indexes on columns used in WHERE, JOIN ON, and ORDER BY clauses.

**Use lazy loading where appropriate** — don't load data until it is needed. Especially important for large object graphs (e.g., ORM relationships).

**Cache hot data** — any data that is read far more often than it is written is a cache candidate. Cache at the database query level (Redis), at the HTTP level (CDN), or at the application level (in-memory LRU).

**Avoid premature optimisation** — write clean code first; then profile. Optimise the actual bottleneck (usually I/O, not CPU), not what you assume is slow.

---

### Functions and classes: best practices

- **Functions should do one thing** — if you need "and" in the function name, split it
- **Keep functions short** — aim for < 20 lines; anything longer is a candidate for extraction
- **Meaningful names** — `calculateMonthlyInterest()` beats `calc()` or `doThing()`
- **Avoid deep nesting** — use guard clauses (early returns) to flatten code
- **Immutability by default** — prefer immutable objects; mutation is a source of hard-to-find bugs
- **Fail fast** — validate inputs at the boundary of your system; don't let bad data propagate deep into business logic

---

### Clean code checklist
- [ ] Does every function/class have a single, clear responsibility?
- [ ] Are there no magic numbers or hardcoded strings? (Use named constants)
- [ ] Are all public interfaces documented?
- [ ] Is the code covered by meaningful tests?
- [ ] Are dependencies injected (not hardcoded)?
- [ ] Is error handling explicit and consistent?
- [ ] Could a new team member understand this code in 5 minutes?

### Key talking point
> "Good low-level design is what makes high-level architecture actually work. A beautifully architected microservices system running N+1 queries in every handler will still crawl under load. Clean, efficient code is not optional — it's the foundation everything else sits on."
